# XCOINGATEWAY — ENVIRONMENT, NETWORK & DEPLOYMENT PROTOCOL
# Source of truth for all infrastructure and deployment decisions

## OVERVIEW

Repository: https://github.com/xpandats/xcoingateway.git
Budget: Zero — all services must use free tiers
Team: 2 people (developer + owner)

This protocol defines the structured, secure, and scalable approach for managing blockchain environments, development lifecycle, API handling, deployment, and infrastructure. The goal is zero-risk deployment, controlled releases, and enterprise-grade reliability.

---

## 1. ENVIRONMENT ARCHITECTURE

### Environments Defined

* LOCAL — Developer machine (your PC)
* DEV — Shared development (can be same as local for 2-person team)
* STAGING — Pre-production testing (MongoDB Atlas free tier)
* PRODUCTION — Live system (VPS + MongoDB Atlas)

### Rules

* Strict isolation between environments
* No shared credentials across environments
* Separate databases, API keys, and wallets per environment
* Environment determined by: NODE_ENV + NETWORK_MODE

## 3. API PROVIDER MANAGEMENT

### MVP API Providers (Zero Budget)

Tron Network:
* Primary: TronGrid (free tier — 15 req/sec, no API key needed for basic, free key for higher limits)
* Secondary: Public Tron full node RPC

Tron Shasta Testnet:
* Primary: TronGrid Shasta endpoint
* Secondary: Public Shasta RPC

### Provider Rotation System

* Automatic failover if primary API fails (timeout, 5xx, rate limited)
* Health check every 30 seconds per provider
* Rate limit tracking (stay under free tier limits)
* Provider status logged (up/down/degraded)
* Alert on failover event

### Chain-Agnostic Adapter Pattern

* Each chain has its own adapter file implementing a common interface
* Adding new chain = adding new adapter + config
* No changes to core business logic when adding chains

### Future Providers (When Scaling)

BNB Chain (BSC):
* Primary: Public RPC (https://bsc-dataseed.binance.org — free, no key)
* Secondary: Ankr free tier

Ethereum:
* Primary: Alchemy free tier (300M compute units/month)
* Secondary: Infura free tier (100k req/day)

## 6. GIT & VERSION CONTROL

### Branch Strategy

* master — production-ready code (protected)
* staging — pre-production testing
* develop — active development
* feature/* — new features (branch from develop)
* hotfix/* — urgent production fixes (branch from master)

### Rules

* No direct commits to master
* Pull request required for all merges
* Code review mandatory (even for 2-person team — review each other's work)
* Squash merge to keep history clean

### Commit Standards

* Clear, descriptive commit messages
* Format: type(scope): description
  - feat(auth): add TOTP 2FA for admin login
  - fix(matching): handle underpayment edge case
  - security(vault): add key zeroing after signing
* No sensitive data in commits (keys, passwords, etc.)
* .gitignore must include: .env*, node_modules, dist, logs

## 7. CI/CD PIPELINE

### Flow

1. Push to feature/* → automated lint + test
2. Merge to develop → DEV build + test
3. Merge to staging → STAGING deployment + integration tests
4. Merge to master → PRODUCTION deployment (manual approval required)

### Automation

* Lint (ESLint) on every push
* Unit tests on every push
* Security audit (npm audit) on every push
* Docker build on merge to staging/master

### Production Deployment

* Docker containers for all services
* docker-compose for orchestration
* Rolling updates (zero downtime)
* Manual approval gate before production deploy

## 12. SECURITY PROTOCOL

### Network Security

* All traffic over HTTPS (TLS 1.3)
* Nginx as reverse proxy + rate limiter
* UFW firewall: whitelist only
* Admin panel: IP whitelisted
* No database ports exposed
* SSH: key-based only, password disabled

### Application Security

* Helmet.js for HTTP security headers
* CORS: strict origin whitelist
* CSP (Content Security Policy) headers
* Rate limiting at Nginx AND application level
* Request size limits
* Input validation on every endpoint (Joi/Zod)
* SQL/NoSQL injection prevention
* XSS prevention

### Secrets Handling

* .env files (never committed to git)
* Production secrets: only on server, not in any repo
* Rotation schedule:
  - JWT secrets: every 90 days
  - Master encryption key: annual (with migration)
  - Merchant API keys: on merchant request
  - Internal service tokens: every 30 days

## 15. FAILURE HANDLING

### Scenarios & Response

| Scenario | Response |
|----------|----------|
| TronGrid API down | Auto-switch to fallback RPC, alert admin |
| MongoDB connection lost | Retry with backoff, queue pending operations, alert |
| Redis down | Graceful degradation, alert (critical) |
| Service crash | Docker auto-restart, startup recovery checks for stuck tx |
| Signing service unresponsive | Queue withdrawal, alert (critical priority) |
| Reconciliation mismatch | Auto-pause withdrawals, alert (critical priority) |

### Principles

* Retry with exponential backoff
* Circuit breakers (stop hammering failed services)
* Graceful degradation (partial functionality > total failure)
* FAIL SAFE: if unsure → block transaction, never assume success
